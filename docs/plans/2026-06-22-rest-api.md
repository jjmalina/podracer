# REST GET API — read access for aggregation apps

**Date:** 2026-06-22
**Status:** Implemented
**Touches:** `podracer/web/app.py`, new `podracer/web/routes/api.py`,
`podracer/db/episodes.py`, `podracer/db/podcasts.py`, `podracer/db/__init__.py`,
`podracer/models.py`, `tests/test_api.py`

## As built (diverges from the design below)

- **A separate `list_episodes`/`count_episodes`, not a widened
  `get_recent_episodes`.** The home feed's `get_recent_episodes` (single
  `topic`, `FeedItem` rows) is left untouched to keep its tested behavior
  stable. The API gets its own `db/episodes.py` primitives that add `podcast_id`
  scoping, multi-tag OR (`t.name IN (...)`), and `has_summary`/`has_transcript`
  flags + optional `summary_data`, via a dynamic `_api_where` builder (the tag
  IN-list count varies, so it can't be the static `_FEED_WHERE` string). The DTO
  is `EpisodeListItem` in `models.py`.
- **Response models live in `web/routes/api.py`, not `models.py`.** They depend
  on `PodcastSummary`/`Chapter`/`Highlight`/`SpeakerIdentification` from
  `summarize.py`; keeping them by the routes avoids importing `summarize` into
  the otherwise dependency-free `models.py`. `Podcast`/`Episode`/`Transcript`
  serialize directly (the detail response subclasses `Episode`).
- **`ApiSummary` is the normalized summary shape** — `summary` / `speakers` /
  `chapters` / `highlights`, where `highlights` is `effective_highlights()` so
  legacy `insights`/`speaker_takes` migrate on read and never appear in the JSON.
- **Tests:** `tests/test_api.py` (19) — DB-layer (`conn` fixture) for
  list/count/tag-OR/scope/pagination, plus `TestClient` route tests for filters,
  422 on bad status/include, the `include=summary` cap drop (200→50),
  legacy-summary migration, ad-speaker filtering, the `local_path`/`artwork_path`
  non-exposure, detail/summary/transcript 404s, and podcasts/tags/version.

### Review hardening (post-review)

A code review of the first cut surfaced six issues, all fixed in the same branch:

- **No server filesystem paths on the wire.** `ApiEpisodeDetail` is now a flat
  explicit model (not an `Episode` subclass), so `local_path` never serializes
  and new episode columns can't auto-leak. Podcasts get an `ApiPodcast` model
  that drops `artwork_path` (kept `feed_url` and the rest).
- **Detail endpoint stopped loading blobs for booleans.** `has_summary`/
  `has_transcript` use new `summary_exists`/`transcript_exists` (`SELECT 1 …
  LIMIT 1`) instead of `get_summary`/`get_transcript`, which loaded the full
  summary JSON and ~200 KB transcript just to check existence.
- **API summaries now strip ad/sponsor speakers** like the web page, via a
  shared `summarize.is_ad_speaker` (de-duplicated from `routes/episodes.py`).
- **`?include` is validated** — anything but `summary` is a 422, not a silent
  no-op at the higher cap.
- **De-duplication:** the tag-filter EXISTS clause is one shared
  `db/podcasts.tag_filter_clause`; the active-job lookup is one
  `db/jobs.ACTIVE_KIND_SUBSELECT` + `get_active_kind`.

## Goal

A read-only JSON API under `/api/v1/...` so programs — not just the HTML UI —
can pull podcasts, the cross-show feed, episode metadata, and summaries. The
driving use case is an **aggregator app running on the same host**: "fetch every
recently-summarized episode tagged *Finance*, pull their summaries, synthesize a
digest, email it." That app needs three things from podracer over HTTP: list
episodes filtered by **tag + status**, read each episode's **summary**, and
(occasionally) read the full **transcript**.

This is the first, GET-only slice of the larger "REST API (JSON) for agent +
script access" roadmap entry. Write endpoints (subscribe, enqueue, ingest),
bearer-token auth, and the MCP wrapper stay deferred — see *Out of scope*.

## Decisions locked in (from review)

1. **Summaries embed inline on the list endpoint via `?include=summary`.** The
   aggregator pulls many summaries per topic run; making it issue N follow-up
   calls is the wrong default. Transcripts stay a separate endpoint regardless
   (they're ~200 KB each — too big to embed in a list).
2. **No auth in this slice.** The aggregator runs on the same host/LAN; the
   existing HTML routes are already open. The bearer-token gate from the
   roadmap lands with the write endpoints, not here.

## What already exists (and we reuse)

- **FastAPI app + router split** (`web/app.py`, `web/routes/*.py`). We add one
  more router, `api.py`, registered alongside the HTML ones. Same
  `Depends(get_db)` per-request connection from `web/deps.py`.
- **`get_recent_episodes(conn, *, limit, offset, subscribed_only, status, topic)`**
  (`db/episodes.py`) — the home-feed query. It **already** filters by `status`
  and by `topic` (a tag name, via an `EXISTS` over `podcast_tags`/`tags`) and
  paginates. This is almost exactly the cross-show list the API needs; the work
  is exposing it as JSON, not writing new SQL. `count_recent_episodes(...)`
  gives the total for pagination metadata.
- **`get_episode` / `get_episodes` / `get_episode_count`** (`db/episodes.py`),
  **`get_podcast` / `get_all_podcasts` / `get_subscribed_podcasts`**
  (`db/podcasts.py`) — direct backing for the per-id and per-podcast endpoints.
- **`get_summary(conn, episode_id) -> SummaryRecord`** (`db/summaries.py`) and
  **`get_transcript(conn, episode_id) -> Transcript`** (`db/transcripts.py`).
- **`get_all_tags(conn, subscribed_only=True)`** (`db/podcasts.py`) backs a
  `GET /api/v1/tags` discovery endpoint so a client can learn the filterable
  values.
- **Pydantic models** (`models.py`): `Podcast` (with `.topics`), `Episode`,
  `FeedItem`, `Transcript`, `SummaryRecord`. FastAPI serializes these to JSON
  for free. `PodcastSummary` (`summarize.py:232`) is the structured summary; the
  DB stores it as a JSON **string** in `summaries.data`, so the API must parse
  it back into structure (see *Summary serialization*).
- **Episode status** is the string lifecycle `pending → downloaded →
  transcribed → summarized`. Tags attach to **podcasts (shows)**, not episodes,
  so an episode's effective topics are its show's topics.

## API surface (`/api/v1`)

All responses are JSON. Read-only — only `GET`.

| Method · Path | Purpose | Key query params |
|---|---|---|
| `GET /api/v1/podcasts` | List shows | `tag`, `subscribed` (bool), `limit`, `offset` |
| `GET /api/v1/podcasts/{id}` | One show (incl. `topics`) | — |
| `GET /api/v1/podcasts/{id}/episodes` | Episodes of one show | `status`, `limit`, `offset` |
| `GET /api/v1/episodes` | **Cross-show feed** — the aggregator's main entry | `tag`, `status`, `subscribed`, `include`, `limit`, `offset` |
| `GET /api/v1/episodes/{id}` | One episode + flags | — |
| `GET /api/v1/episodes/{id}/summary` | The `PodcastSummary` (structured) | — |
| `GET /api/v1/episodes/{id}/transcript` | Full transcript text (large) | — |
| `GET /api/v1/tags` | Filterable tag values | `subscribed` (bool) |
| `GET /api/v1/version` | `{podracer_version, schema_version}` | — |

### `GET /api/v1/episodes` — the core endpoint

This is the feed, exposed for machines. It mirrors the existing
`get_recent_episodes` filters and adds inline summaries:

```
GET /api/v1/episodes?tag=Finance&status=summarized&include=summary&limit=50
```

```jsonc
{
  "items": [
    {
      "id": 42,
      "podcast_id": 7,
      "podcast_title": "Odd Lots",
      "title": "The copper squeeze nobody saw coming",
      "published_at": "2026-06-21T13:00:00",
      "status": "summarized",
      "duration_seconds": 3120,
      "has_summary": true,
      "has_transcript": true,
      "active_job": null,
      "summary": {            // present only when ?include=summary
        "summary": "…",
        "speakers": [ … ],
        "chapters": [ … ],
        "highlights": [ … ]
      }
    }
  ],
  "total": 137,
  "limit": 50,
  "offset": 0
}
```

- `tag` → passed through as `topic` to `get_recent_episodes` (filters by the
  show's tag). Omit / `all` = no tag filter.
- `status` → one of `pending|downloaded|transcribed|summarized`, or omit / `all`.
- `subscribed` (default `true`) → maps to `subscribed_only`.
- `include=summary` → embed `PodcastSummary` per row. Without it, each row still
  carries `has_summary` so the client can decide whether to fetch. **Page size is
  capped lower when embedding** — `MAX_LIMIT` stays 200 for metadata-only, but
  `include=summary` caps the effective limit at 50 so one response stays ~MB
  rather than tens of MB.
- `tag` → **repeatable, OR-semantics.** `?tag=Finance&tag=Macro` returns episodes
  whose show carries *Finance* OR *Macro*. A single `?tag=Finance` is the
  one-element case. FastAPI binds a repeated query param to `tag: list[str] |
  None`. Omit / `["all"]` = no tag filter.

### Summary serialization

`summaries.data` is a JSON string of `PodcastSummary`. The summary endpoints and
the inline `include=summary` path must `PodcastSummary.model_validate_json(rec.data)`
and emit `summary.effective_highlights()`-style structure (so legacy
`insights`/`speaker_takes` rows migrate to `highlights` on read, exactly as the
HTML episode page does). Don't pass the raw DB string through — the response
contract is the structured object, not "a string that happens to be JSON".

### Response models

Add purpose-built response models to `models.py` rather than leaking DB rows:

- `EpisodeSummaryItem(FeedItem-ish + has_summary, has_transcript, active_job,
  summary: PodcastSummary | None)` — the `/episodes` row. Extend `FeedItem` or
  add a sibling; the existing feed query already returns `active_kind`, so
  `has_*` flags are cheap LEFT-JOIN existence checks added to the same query (no
  N+1).
- `EpisodeDetail` — `Episode` plus `has_summary`, `has_transcript`,
  `active_job`, and the show's `topics`.
- `Paginated[T]` envelope `{items, total, limit, offset}` (a small generic or
  per-endpoint model) so every list endpoint reports `total` for paging.

Transcript and per-episode summary reuse `Transcript` / `PodcastSummary`
directly.

## Data layer

Mostly reuse. Two small additions to `db/episodes.py`:

1. Extend the feed query (or add an API-specific variant) to also return
   `has_summary` / `has_transcript` existence flags and — when summaries are
   requested — join `summaries.data`. Keep it one query to preserve the
   no-N+1 property the feed plan established.
2. **Widen tag filtering from one to many.** `get_recent_episodes`'s `topic:
   str` predicate (an `EXISTS` with `t.name = ?`) becomes a `tags: list[str] |
   None` predicate (`EXISTS (… WHERE t.name IN (?, ?, …))`) — OR-semantics, a
   show matches if it carries any requested tag. Build the placeholders from the
   list length and pass the names as params (never string-interpolate). A single
   tag is the one-element case, so the existing feed route keeps working by
   passing `[topic]` (or keep a thin `topic=` shim). `count_recent_episodes`
   takes the same widened predicate.
3. `get_podcasts(conn, *, tags=None, subscribed_only=True, limit, offset)` in
   `db/podcasts.py` for the filtered podcast list (today's `get_all_podcasts` /
   `get_subscribed_podcasts` don't take a tag filter or paginate). Reuse the same
   `IN (...)` `EXISTS (podcast_tags JOIN tags)` predicate.

Everything else (`get_episode`, `get_summary`, `get_transcript`,
`get_all_tags`, `get_episode_count`) is called as-is. Re-export any new
functions from `db/__init__.py`.

## Route layer

New `podracer/web/routes/api.py`:

```python
router = APIRouter(prefix="/api/v1", tags=["api"])

SCHEMA_VERSION = "v1"    # bumped on a breaking API-shape change
MAX_LIMIT = 200          # hard cap, metadata-only
SUMMARY_MAX_LIMIT = 50   # hard cap when ?include=summary embeds full summaries
DEFAULT_LIMIT = 50
STATUSES = {"pending", "downloaded", "transcribed", "summarized"}

@router.get("/episodes")
def list_episodes(
    db: sqlite3.Connection = Depends(get_db),
    tag: list[str] | None = Query(None),  # repeatable: ?tag=Finance&tag=Macro (OR)
    status: str | None = None,
    subscribed: bool = True,
    include: str | None = None,           # "summary"
    limit: int = DEFAULT_LIMIT,
    offset: int = 0,
) -> PaginatedEpisodes:
    cap = SUMMARY_MAX_LIMIT if include == "summary" else MAX_LIMIT
    limit = max(1, min(limit, cap))
    ...
```

- Register first-or-last in `web/app.py` via `app.include_router(api_router)`.
  Order doesn't matter — the `/api/v1` prefix can't collide with the HTML routes.
- Validate `status` against `STATUSES` (else `422`), `limit` clamped to
  `MAX_LIMIT`. Unknown `tag` → empty result set, not an error.
- `404` for missing episode/podcast/summary/transcript ids, with a JSON body
  `{"detail": "..."}` (FastAPI default).
- **OpenAPI/Swagger** at `/api/v1/docs` comes free. Tag these routes
  (`tags=["api"]`) and consider `include_in_schema=False` on the HTML routes so
  the generated schema stays API-only.

## Filtering + pagination (the contract)

- `?limit=` default 50, hard cap 200; `?offset=` for paging. Every list
  response includes `total` so a client can loop until `offset + len(items) >=
  total`.
- `?status=` and `?tag=` are the two filters the aggregator drives. Both reuse
  the existing feed predicates — no new query semantics to test from scratch.
- **`?since=<iso>`** for incremental polling (episodes with
  `COALESCE(published_at, created_at) > since`) is the obvious next filter for a
  scheduled aggregator that doesn't want to re-pull the world. Cheap to add to
  the same WHERE; include in v1 if the aggregator is built in the same pass,
  otherwise fast-follow.

## Testing

New `tests/test_api.py` (FastAPI `TestClient`, in-memory SQLite fixture like
`tests/test_feed.py`):

- `GET /api/v1/episodes` returns 200, correct shape, `total` accurate.
- `status` filter: only matching episodes; bad status → 422.
- `tag` filter: episode under a show tagged *Finance* appears; untagged show's
  episode excluded; unknown tag → empty `items`, `total: 0`.
- `subscribed=false` includes unsubscribed shows' episodes.
- `include=summary`: row carries parsed `summary` with `highlights`; a legacy
  insights/takes summary migrates on read; without the param, `summary` is
  absent but `has_summary` is correct.
- Pagination: disjoint, gap-free windows across pages; `limit` clamped to 200.
- `GET /episodes/{id}/summary` and `/transcript`: 200 with structure / text;
  404 when absent.
- `GET /podcasts?tag=…&subscribed=…`, `/tags`, `/version` smoke tests.

Run `.venv/bin/ruff check podracer/` and `.venv/bin/ty check podracer/` before
committing. No nested imports — new helpers live at module top in `db/` and the
route module.

## Resolved (was open)

- **`include=summary` page size — capped.** `MAX_LIMIT` stays 200 for
  metadata-only listings; `include=summary` lowers the effective cap to
  `SUMMARY_MAX_LIMIT = 50` so an embedded-summary response stays ~MB.
- **Multi-tag — supported, OR-semantics.** `?tag=` is repeatable; a show matches
  if it carries any requested tag (`t.name IN (...)`). The feed's single-`topic`
  predicate is widened to a `tags` list in `db/episodes.py`.
- **`schema_version` — a code constant.** `SCHEMA_VERSION = "v1"` lives in the
  API router module and is returned by `GET /api/v1/version` alongside the
  package version. It's the **API contract** version (bumped on a breaking
  response-shape change), independent of the DB migration state — no `config`
  row, no coupling to `_migrate`.
