# Home feed — recent episodes across all shows

**Date:** 2026-06-15
**Status:** Implemented (branch `feed-page`)
**Touches:** `podracer/db/episodes.py`, `podracer/web/routes/`, `podracer/web/templates/`, `podracer/web/static/style.css`, `podracer/db/connection.py`

## As built (diverges from the proposal below)

The proposal below is the original design; a few things changed during the build:

- **No subscribed/all toggle — the feed is always subscribed-only.** The
  proposed `?scope=all` control was dropped: the feed is your subscriptions,
  full stop. (`get_recent_episodes` keeps a `subscribed_only` arg for generality,
  but the route always passes `True`.)
- **Status filter, defaulting to `summarized`.** Instead of a scope toggle, the
  feed has status chips (`summarized` · `pending` · `all`) and defaults to
  **summarized** — the episodes that are actually ready to read.
  `get_recent_episodes`/`count_recent_episodes` take a `status` arg (`None`/`all`
  = no filter), so other statuses remain reachable by URL even without a chip.
- **Timestamps become absolute dates past ~3 days.** `just now` / `Nm` / `Nh` /
  `yesterday` / `2–3d ago`, then a real date (`Jun 5`, or `Jun 5, 2024` for a
  prior year).
- **Per-row badge rules.** The status badge is suppressed when a specific status
  is selected (every row shares it) and shown only in the `all` view. An
  in-flight job renders a `…running` badge — *except* on a `summarized` episode,
  where a leftover queued/running job is stale and must not contradict the
  terminal status.
- **Tests:** `tests/test_feed.py` (10) — ordering + NULL-date fallback, the
  `subscribed_only` and `status` filters, disjoint/complete pagination windows,
  active-job surfacing, the relative-time→date thresholds, and the
  summarized-episode stale-badge regression.

## Goal

A landing page at `/` that shows a **reverse-chronological, paginated list of
recently published episodes across every subscribed show** — the "what's new in
my subscriptions" inbox. Today the web UI is entirely *per-podcast*: you land on
`/podcasts` (an alphabetical grid of shows) and have to click into each one to
see its episodes. There is no single place that answers "what showed up
recently across everything I follow."

This is a **read/presentation feature only.** Ingestion, the subscription
watermark, and the transcribe→summarize pipeline already exist (`subscribe`
CLI + `worker._sync_feeds()` + `find_new_episodes()`). Nothing about how
episodes get *into* the DB changes. We are adding one cross-show query, one
route, one template, and supporting CSS.

## What already exists (and we reuse)

- **`episodes` table** (`db/connection.py:28-43`) — has `published_at`,
  `created_at`, `status` (`pending`→`downloaded`→`transcribed`→`summarized`),
  and `podcast_id`.
- **`published_at` is ISO-8601** — `feed.py:32` stores
  `datetime(*published[:6]).isoformat()` → e.g. `2026-06-15T14:30:00`. A lexical
  `ORDER BY ... DESC` is therefore already chronological. It can be `NULL` when a
  feed omits the date, so we coalesce to `created_at` (`datetime('now')`).
- **Artwork endpoint** — `GET /podcasts/{id}/artwork` (`routes/podcasts.py:50`)
  already serves cached covers + deterministic placeholders. Feed rows reuse it;
  no new image plumbing.
- **Status data** — `episodes.status` plus the active-job lookup pattern from
  `routes/episodes.py:119-123` (`SELECT kind, status FROM jobs WHERE episode_id=?
  AND status IN ('queued','running')`).
- **Design system** — instrument-panel aesthetic, amber accent, LED dots,
  `IBM Plex` / `Space Grotesk` type, `.cover` tiles, `.eyebrow`, `.badge`
  (`static/style.css`, see `podcasts/list.html` for the idiom to mirror).
- **htmx 2.0** is already loaded in `base.html:12` — available for "Load more"
  if we want it, but the v1 plan uses plain server-rendered page links.

## Scope decisions

1. **Rows are episodes, not shows.** "List of podcasts in reverse chronological
   order" is read as *podcast episodes* — matching the timeline mock that was
   chosen (one row per episode: show name, episode title, relative time,
   status). Ordering the *shows* grid by recency is a trivial separate change
   (`get_all_podcasts` `ORDER BY title` → a recency column) and is noted under
   follow-ups, not built here.

2. **Default scope = subscribed shows.** This is a *subscription* feed, so by
   default it lists episodes from podcasts where `subscribed = 1`. A toggle
   (`?scope=all`) shows the whole library (covers one-off downloads / shows
   added but not subscribed). Open question below if you'd rather default to all.

3. **Sort key = `COALESCE(published_at, created_at) DESC, id DESC`.** Newest
   first; `id DESC` is a stable tiebreaker so pagination can't drop or repeat a
   row when two episodes share a timestamp.

4. **Status is a badge, not a progress bar.** `jobs` has no percent column
   (`connection.py:63-75`), so each row gets a labelled badge/LED — `Summarized`
   / `Transcribed` / `Downloaded` / `Queued` / `Running` / `Pending` — derived
   from `episodes.status` and any active job. No fake "40%".

## Data layer

One new function in `db/episodes.py`, plus its re-export in `db/__init__.py`.
Returns a lightweight row object (episode fields + the show's title and
podcast_id + active-job kind) so the template needs no per-row follow-up query —
avoiding the N+1 that a naive loop would create.

```python
# db/episodes.py
def get_recent_episodes(
    conn: sqlite3.Connection,
    *,
    limit: int,
    offset: int = 0,
    subscribed_only: bool = True,
) -> list[FeedItem]:
    """Episodes across all (optionally only subscribed) shows, newest first.

    Sort: COALESCE(published_at, created_at) DESC, e.id DESC.
    Joins podcasts for the show title + a single active job kind so the
    feed template renders without per-row queries.
    """

def count_recent_episodes(
    conn: sqlite3.Connection, *, subscribed_only: bool = True,
) -> int:
    """Total matching rows — for page count. Same WHERE clause as above."""
```

Sketch of the core query (LEFT JOIN to a *single* active job per episode so a
row still appears when no job exists):

```sql
SELECT
    e.id, e.podcast_id, e.title, e.published_at, e.created_at,
    e.status, e.duration_seconds,
    p.title AS podcast_title,
    (SELECT j.kind FROM jobs j
      WHERE j.episode_id = e.id AND j.status IN ('queued','running')
      ORDER BY j.id LIMIT 1) AS active_kind
FROM episodes e
JOIN podcasts p ON p.id = e.podcast_id
WHERE (:subscribed_only = 0 OR p.subscribed = 1)
ORDER BY COALESCE(e.published_at, e.created_at) DESC, e.id DESC
LIMIT :limit OFFSET :offset;
```

`FeedItem` is a small Pydantic model in `models.py` (episode subset +
`podcast_title` + `active_kind`), or a `TypedDict` like `ChapterBucket` in
`routes/episodes.py:19`. Pydantic is the house style — use that.

**Index (recommended).** Add to `SCHEMA` in `connection.py` and to `_migrate`
so existing DBs get it, matching the `idx_jobs_*` pattern:

```sql
CREATE INDEX IF NOT EXISTS idx_episodes_recency
    ON episodes(COALESCE(published_at, created_at) DESC, id DESC);
```

For a personal archive (hundreds–low thousands of rows) it's not strictly
required, but it's cheap and keeps `OFFSET` paging on the deep pages honest.

## Route layer

New `podracer/web/routes/feed.py` (mirrors the per-area router split:
podcasts / episodes / search / jobs):

```python
PAGE_SIZE = 30

@router.get("/")
def feed(request, page: int = 1, scope: str = "subscribed",
         db = Depends(get_db)):
    subscribed_only = scope != "all"
    page = max(1, page)
    total = count_recent_episodes(db, subscribed_only=subscribed_only)
    items = get_recent_episodes(
        db, limit=PAGE_SIZE, offset=(page - 1) * PAGE_SIZE,
        subscribed_only=subscribed_only,
    )
    pages = max(1, math.ceil(total / PAGE_SIZE))
    return templates.TemplateResponse(request, "feed/list.html", {
        "request": request, "items": items, "page": page, "pages": pages,
        "scope": "all" if subscribed_only is False else "subscribed",
        "relative_time": relative_time, "format_duration": _format_duration,
    })
```

Wiring changes:
- **`routes/podcasts.py:34-36`** — `GET /` currently `RedirectResponse(url="/podcasts")`.
  Remove that handler; `/` is now the feed. `/podcasts` keeps its grid.
- **`web/app.py:78`** — `app.include_router(feed_router)` (register first).
- **`base.html:44-48`** — add a `Feed` nav item (first), and consider repointing
  the brand link from `/podcasts` to `/`.

## Pagination

Server-side `LIMIT/OFFSET`, classic page links — no JS required, shareable URLs
(`/?page=3`, `/?scope=all&page=2`). `PAGE_SIZE = 30` (tunable).

- Compute `pages = ceil(total / PAGE_SIZE)` from `count_recent_episodes`.
- Render **‹ Newer** (disabled on page 1) / **Page N of M** / **Older ›**
  (disabled on the last page). Preserve `scope` in the links.
- `id DESC` tiebreaker (above) guarantees a stable window across pages.
- **Stretch (optional):** htmx "Load more" that appends the next page into the
  list (`hx-get="/?page=N"`, swap a `_rows.html` partial). htmx is already
  loaded; defer to keep v1 simple and bookmarkable.

## Relative-time helper

"2h ago" / "yesterday" / "3d ago" from the ISO timestamp. New small helper
(e.g. `web/format.py` or a Jinja filter registered next to `linkify` in
`app.py:75`). Watch the format mismatch: `published_at` is `...T...` (naive
local, from `datetime(...).isoformat()`), `created_at` is `datetime('now')`
(`YYYY-MM-DD HH:MM:SS`, **UTC**). Parse both, treat as UTC-ish, clamp negative
deltas to "just now". Falls back to the raw date string if parsing fails.

## Edge cases / risks

- **NULL `published_at`** — handled by the `COALESCE`. Verified `feed.py` can
  emit `None` (`_parse_date` returns `None`).
- **Timestamp format skew** (`T` vs space, local vs UTC) — only affects ordering
  when two rows tie on the date prefix; the `id DESC` tiebreaker keeps paging
  stable regardless. Relative-time helper must tolerate both formats.
- **N+1 queries** — avoided by doing the show-title join + active-job subselect
  inside the one query.
- **Deep `OFFSET`** — fine at personal scale; the recency index covers it.
  Keyset pagination is the upgrade path if the archive ever gets huge (noted,
  not built).
- **No subscriptions yet** — empty state nudges toward `?scope=all` / subscribe.

## Testing

New `tests/test_feed.py` (style of `tests/test_subscription.py`):
- Ordering: insert episodes with mixed `published_at` (incl. one `NULL`) across
  two shows → assert `get_recent_episodes` returns strict newest-first order and
  the `NULL`-date row sorts by its `created_at`.
- `subscribed_only`: an episode under an unsubscribed show is excluded by
  default, included with `subscribed_only=False`.
- Pagination: `count_recent_episodes` + `limit/offset` produce non-overlapping,
  gap-free windows; `id DESC` tiebreak holds when timestamps collide.
- Route smoke test (FastAPI `TestClient`): `GET /` is 200 and renders an
  episode title; `GET /?page=2` and `?scope=all` work; pager disables correctly
  at the ends.

## Out of scope / follow-ups

- **Shows-by-recency.** Re-sorting the `/podcasts` grid by latest-episode
  recency instead of `title` — a one-line query change; do it separately if
  wanted.
- **Unread / seen tracking.** A real "inbox" would track which episodes you've
  opened. Needs a new column/table; defer.
- **Per-section feeds.** "Only summarized", "only downloaded" filters beyond the
  subscribed/all toggle.
- **Keyset pagination.** Only if the archive outgrows `OFFSET`.
